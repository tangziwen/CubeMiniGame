# CubeGame Editor Design

## Design Intent

编辑器是游戏的一部分，不是外部开发工具。它让玩家暂时从游玩状态切换到创造状态，在同一个世界里修改地形、铁路和未来的 gameplay 对象。

编辑器的目标不是提供大量复杂面板，而是让玩家清楚知道：现在正在编辑什么、下一次点击会做什么、当前选中的对象是什么、系统为什么拒绝某个操作。

## Experience

- 编辑行为应尽量像直接操作世界。
- 工具切换应轻，不需要层层进入面板。
- 检查对象应是非模态的；查看、选择、重命名不应该阻断其他编辑流程。
- 只有真正需要抢占点击的编辑动作才应该表现为模态。
- 失败反馈应该明确，无效操作不应静默失败。
- 当 UI 变拥挤时，优先使用独立浮动窗口，而不是深层嵌套面板。

## Design

### In-world Editing

编辑器的核心机制是让玩家直接修改当前世界。UI 负责选择编辑意图，真正的编辑对象仍然是世界中的地形、铁路和 gameplay 对象。

### Terrain Editing

地形编辑是编辑器的基础工作流之一。它包括球形/盒形的挖去与抬升，以及按材质绘制地形。地形工具应以光标所在世界位置为目标，让玩家感觉是在直接雕刻当前世界。

### Lightweight Tool State

编辑器需要知道当前模块和当前动作，但这些状态只表达玩家正在做什么，不应该成为 gameplay 对象本身的一部分。

### Editor Modality

编辑模态是编辑器的核心逻辑：它说明当前玩家的世界点击会被哪个编辑动作解释。比如“选择对象”“添加轨道”“删除站点”“添加线路控制点”都可能看起来是一次点击，但它们在不同模态下代表完全不同的设计意图。

模态不等于窗口是否打开，也不等于检查面板是否存在。模态只应该描述会抢占世界点击或快捷键的主动编辑行为；检查、查看、重命名这类观察和属性操作默认是非模态的。

### Non-modal Inspection

检查对象是一种持续观察和属性查看能力。它可以辅助选择、重命名和高亮，但默认不应该把玩家锁进独立流程。

### Explicit Feedback

当编辑动作不能完成时，系统应该把失败作为玩家体验的一部分表达出来，而不是让玩家猜测是否点错或系统失效。

### Editor Feedback

参数面板、预览、提示、删除按钮和可点击 billboards 都是编辑反馈机制。它们服务当前编辑意图，但不应该被当成独立 gameplay 对象。

## Game Concepts

- 编辑器：玩家进入创造/调整状态后的整体工作台。
- 编辑模块：地形、铁路、放置等不同编辑领域。
- 编辑动作：某个模块下当前选中的具体操作。
- 编辑模态：当前会解释世界点击或快捷键的主动编辑状态。
- 地形编辑：对世界基础地形进行挖去、抬升或材质绘制。
- 世界点击：玩家对场景中对象或地形的直接操作。
- 检查目标：当前被玩家查看、选择、重命名或高亮的对象。

## Concept Code Anchors

- 编辑器 -> `EditorPanel`
- 编辑状态 -> `EditorState`
- 编辑模态 -> `EditorState`
- 地形编辑 -> `TerrainEditingController`
- 检查目标 -> `InspectPanel`

## Critical Code Anchors

- `EditorController`：编辑状态和领域控制器的协调入口。
- `TerrainEditorImGuiView` / `RailEditorImGuiView`：编辑 UI 表现锚点，不是 gameplay concept。
- `TerrainEditingController` / `RailEditingController`：把编辑意图转成领域动作的技术锚点。

## Design Boundaries

- UI 负责选择和展示，不负责拥有 gameplay 真相。
- 编辑状态属于编辑器，不应该写进铁路线路、列车或其他 gameplay 数据模型。
- 编辑模态只用于主动编辑行为；检查、查看和重命名默认不应被做成模态。
- 世界点击属于“玩家直接操作世界”的概念，不应该被 UI 绘制细节吞掉。
- `InspectPanel` 默认是观察和属性层，不是强制模态流程。
- 具体 gameplay 修改应落到对应 gameplay 系统，而不是停留在 UI 层。
- 参数面板、预览和按钮属于编辑反馈机制，不作为独立 gameplay concept 维护。
- 当前不把编辑器做成完整商用关卡编辑器。
- Undo/redo 是独立主题；存档/读档属于世界持久化，不由编辑模态或 UI 状态拥有。
- 不为了“框架完整”而牺牲当前编辑体验的简单直接。
