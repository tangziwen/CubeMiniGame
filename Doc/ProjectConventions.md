# Project Conventions

本文记录项目级结构约束、阅读入口和代码风格。它是人类半手工维护的约定入口，不替代源码或完整文件索引。

## Navigation

- 引擎模块导航入口是 `Doc/EngineCodeMap/README.md`；Gameplay/编辑器意图入口是 `Doc/GameplayIntent/README.md`。
- CodeMap 和 GameplayIntent 只记录能力、入口、契约、设计意图和已知坑。
- 需要确认实现时，用 `rg` 搜索源码中的文件名、模块名或功能关键词作为入口，最终判断以源码为准。

## Dependency Direction

- Application 依赖 Engine API；Engine 不依赖具体游戏业务。
- Engine 可以调度子系统，但子系统之间应尽量通过明确接口协作。
- 低层工具、数学、文件和基础类型不应依赖渲染、场景或应用层。
- 渲染对象可以依赖资源和后端抽象；避免绕过渲染调度访问平台细节。
- 新增全局单例前先确认是否已有所有者或生命周期入口可以承载。

## Runtime Flow

1. `main()` 调用 `Engine::preSetting()` 完成基础路径、随机数、崩溃处理等预设置。
2. `Engine::run()` 设置应用入口，加载配置，创建窗口后端并进入窗口主循环。
3. 启动阶段初始化日志、渲染信息、场景、资源、脚本、GUI、音频等核心系统。
4. 每帧更新逻辑、事件、物理、脚本/业务入口和场景遍历。
5. 渲染阶段由 `GraphicsRenderer` 等高层渲染系统组织提交，最终交给当前渲染后端执行。

## Change Guidance

- 局部功能优先落在已有子系统边界内；跨模块改动先明确所有权、生命周期和调用方向。
- 渲染后端改动保留后端抽象边界，避免把 Vulkan 细节扩散到无关模块。
- 文档与实现不一致时以源码为准，修改后顺手校正文档入口。
- 历史代码不要求批量清洗；局部修改时优先保持所在文件的一致性。

## Module And File Layout

- 目录即模块。新类型、新功能或文档入口优先放在所属概念目录下。
- 模块内部可以有私有 helper；跨模块依赖应指向模块对外入口。
- 通常一类一文件，文件名与主类型名一致，例如 `RailSystem.h` / `RailSystem.cpp`。
- `helpers`、`info`、`config`、纯数据结构集合、很小的内部辅助类型等可以合并在同一个文件里，但不要让文件变成无边界的杂物箱。
- 新增、删除或挪动参与编译的 C++ 文件时，同步更新对应的 `CMakeLists.txt`。

## C++ Baseline

- 使用 C++17。
- 保持实现直接、可读，避免为局部问题引入复杂模板或过度抽象。
- 资源所有权要明确：新代码优先使用 RAII；必须使用裸指针时，在命名、生命周期或调用点上让所有权关系清楚。
- 容器、字符串和算法优先使用标准库；不要手写标准库已经可靠提供的基础能力。

## Object Construction

- `Node` 派生对象通常使用两段式构造，并提供专门的 `create` 方法。
- 场景对象由场景树或对应系统管理特殊生命周期，一般不使用共享智能指针表达所有权。
- `data`、`info`、`config` 等简单辅助类不适用 `Node` 两段式构造规则。
- 数据配置型辅助类尽可能避免堆分配，优先作为值对象、成员对象或局部对象使用。

## Formatting

- 使用 Allman 大括号风格；仓库已有 `format.astylerc` 作为格式化锚点。
- 缩进使用 4 个空格；编辑旧文件时可跟随该文件现有缩进。
- 不做无关格式化；修改一段代码时只整理相关区域。
- 控制流关键字后保留空格，例如 `if (...)`、`for (...)`、`while (...)`。

## Naming

- 命名优先跟随所在模块的现有风格。
- 类型名使用 PascalCase，例如 `SceneMgr`、`RenderBackEndBase`。
- 函数和局部变量使用 lowerCamelCase。
- 成员变量使用 `m_` 前缀。
- 常量、宏和编译开关使用大写下划线；宏只在确有必要时使用。
- 优先使用业界通行缩写，例如 `info`、`config`、`mgr`、`usr`、`curr`、`prev`，避免链式调用名称过长。
- 管理对象生命周期、注册表、查找表或跨帧状态的类，通常使用 `Mgr` 或 `System` 后缀，并作为单例或全局服务入口。
- 每帧调用的函数必须使用 `update` 或 `tick` 作为前缀，例如 `updateTrains()`、`tickSimulation()`。
- 事件处理函数使用 `on` 开头，写出事件来源或语义，例如 `onMouseClick()`、`onWindowResize()`。
- 全局变量和自由函数使用 `g_` 前缀，并使用 PascalCase，例如 `g_CurrScene`、`g_GetCurrScene()`。
- 序列化反序列化或者类似保存/读取之类写入文件的操作统一用'serialize'/'unserialize'前缀
- 布尔查询谓词优先使用能表达语义的前缀：状态用 `is`，拥有关系用 `has`，能力用 `can`，策略判断用 `should`，例如 `isVisible()`、`hasSelection()`、`canUndo()`、`shouldRebuildMesh()`。
- 查找失败合法时优先用 `find`；对象必须存在时优先用 `get`。构造新对象或资源时按语义选择 `create`、`build` 或 `load`。

## Headers

- 头文件保持自包含：依赖的类型、标准库头和前置声明应能支持单独包含。
- 优先使用前置声明减少头文件依赖；需要完整类型时再包含头文件。
- include 顺序建议为：对应头、项目头、第三方头、标准库头；编辑旧文件时以局部一致性为先。
- 新头文件使用现有 include guard 风格。

## Error Handling

- 对外部输入、文件、JSON、资源加载和图形 API 调用做显式失败处理。
- 失败路径留下足够日志，方便定位资源路径、接口返回值或配置字段。
- 不吞掉错误；无法恢复时尽早返回或中止当前操作。

## Comments

- 默认倾向于不加注释，让命名、类型和结构自己解释代码。
- 成段算法、复杂生命周期、跨模块约束或用户明确要求记录时，可以加简短注释说明意图。
- 不为显而易见的赋值、分支和包装函数添加注释；临时调试注释提交前移除。
